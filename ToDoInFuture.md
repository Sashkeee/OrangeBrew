# 🚀 OrangeBrew: План развития отказоустойчивости и масштабируемости

Этот документ содержит техническое описание архитектурных улучшений, необходимых для превращения OrangeBrew в промышленно-надежную систему (Resilience & Scalability).

---

## 🛡 1. Механизм восстановления сессий (Session Persistence)

**Проблема:** При перезагрузке сервера или критическом сбое все текущие процессы (затирание, варка) прерываются, а их состояние в памяти теряется.

### Шаг 1: Изменение БД
Добавить в таблицу `brew_sessions` поле для хранения JSON-слепка состояния.
```sql
ALTER TABLE brew_sessions ADD COLUMN last_state_json TEXT;
```

### Шаг 2: Реализация снапшотов (`ProcessManager.js`)
Внедрить периодическое сохранение состояния в методе `updateLoop`:
- Каждые 10 секунд вызывается метод `saveSnapshot()`.
- Состояние включает: `currentStepIndex`, `remainingTime`, `stepPhase`, `status`.

### Шаг 3: Автоматическое восстановление (`server.js`)
При старте бэкенда:
1. Запрос в БД: `SELECT * FROM brew_sessions WHERE status = 'active'`.
2. Для каждой найденной сессии:
   - Инициализация нового `ProcessManager`.
   - Загрузка данных из `last_state_json`.
   - Если задержка (текущее время минус время снапшота) < 5 минут — автоматический запуск PID и таймеров.
   - Если задержка > 5 минут — восстановление в режиме `PAUSED` с уведомлением в Telegram.

## ⚡ 2. Масштабирование на 100+ пользователей

### Переход на внешнюю БД
- Замена `SQLite` на `PostgreSQL` для исключения блокировок базы при параллельной записи логов от сотен датчиков.

### Интеграция Redis
- Использовать Redis для хранения "горячих" состояний `ProcessManager` вместо локальной памяти (Map).
- Это позволит запускать несколько инстансов сервера (Cluster Mode) за балансировщиком, где каждый инстанс будет иметь доступ к общему состоянию.

### Оптимизация Event Loop
- Вынос математических расчетов (PID, фильтр Калмана) в `Worker Threads`, чтобы интенсивный поток данных от датчиков не блокировал обработку HTTP-запросов.

## 📟 3. Аппаратная безопасность (Hardware Watchdog)

**Проблема:** Если сервер зависнет, ТЭН может остаться включенным на 100%.

**Решение:**
- Внедрение механизма "Heartbeat" в прошивку ESP32.
- Если ESP32 не получает команд от сервера более 60 секунд, она автоматически сбрасывает мощность ТЭНа (`Heater`) и Насоса (`Pump`) в 0.
- Визуальная индикация потери связи на устройстве.

---

## 📈 4. Мониторинг и метрики (Observability)

- Интеграция `Prometheus` для сбора метрик: количество активных сессий, нагрузка на CPU от PID-циклов, объем записей в БД.
- Визуализация в `Grafana` для отслеживания стабильности системы под нагрузкой.

## 📋 5. Swagger / OpenAPI документация

Добавить интерактивную документацию API через `swagger-ui-express` + `swagger-jsdoc`.

- Установить: `npm install swagger-ui-express swagger-jsdoc`
- Подключить в `server.js` → доступна по `/api/docs`
- Добавить JSDoc комментарии в каждый роут (`routes/*.js`)
- Польза: QA может тестировать прямо из браузера, Postman коллекцию можно сгенерировать из OpenAPI spec автоматически

## 🧠 Приложение: Анализ выбора технологии (Redis vs SQLite Snapshots)

При выборе пути реализации отказоустойчивости (Resilience) мы сравнили три подхода:

| Критерий | Текущий (Memory Map) | SQLite Snapshots | Redis (Distributed Cache) |
| :--- | :--- | :--- | :--- |
| **Сложность** | Низкая (уже внедрено) | Низкая (использует существующую БД) | Высокая (нужен отдельный сервис) |
| **Отказоустойчивость** | ❌ Нет (данные в RAM) | ✅ Да (состояние на диске) | ✅ Да (состояние в памяти Redis) |
| **Портативность** | ✅ Высокая | ✅ Высокая (все в одном файле) | ❌ Низкая (нужна установка Redis) |
| **Масштабируемость** | ❌ Нет | ⚠️ Ограничена I/O диска | ✅ Безгранично (Cluster Mode) |

### Рекомендация Senior Developer:
1. **Для домашнего использования / Self-hosting:** Использовать **SQLite Snapshots**. Это обеспечит выживание процесса при перезагрузке сервера без усложнения установки для пользователя.
2. **Для облачного решения (SaaS) на 500+ портов:** Переход на **Redis + PostgreSQL**. Это единственный способ обеспечить работу нескольких инстансов бэкенда с общими данными.
