
# 🍺 Анализ проекта OrangeBrew

## 1. Обзор

**OrangeBrew** — это полнофункциональное веб-приложение для управления процессами пивоварения и дистилляции. Оно использует бэкенд на **Node.js** с **Express** и **WebSockets**, а также фронтенд на **React 19** с использованием **Vite**. Проект выполнен в виде монорепозитория.

*   **Основная логика**: Кастомная реализация PID-контроллера, симуляция оборудования (Mock).
*   **Хранение данных**: SQLite (`sql.js`) для портативности.
*   **Коммуникация**: WebSocket для данных датчиков в реальном времени, REST API для CRUD-операций.

## 2. Сильные стороны

### 🏗 Архитектура и качество кода
*   **Современный стек**: Использование новейших возможностей React 19, быстрая сборка Vite и ES Modules на бэкенде.
*   **Разделение ответственности**: Четкое разделение логики оборудования (`serial/`, `pid/`), слоя данных (`db/`) и маршрутов API.
*   **Абстракция оборудования**: Класс `MockSerial` — отличное решение, позволяющее полноценно разрабатывать и тестировать без физического оборудования.
*   **Real-time возможности**: Надежная реализация WebSocket обеспечивает мгновенное обновление данных датчиков в UI.

### 🧪 Обеспечение качества
*   **Культура тестирования**: Высококачественные тесты с использованием `Vitest` как для бэкенда (логика, PID), так и для фронтенда (компоненты).
*   **Устойчивость**: Бэкенд корректно обрабатывает завершение работы, обеспечивая целостность базы данных и безопасность оборудования.

### 🎨 Опыт пользователей (Frontend)
*   **Визуальная обратная связь**: Хорошее использование `Recharts` для визуализации процессов и `Framer Motion` для анимаций интерфейса.
*   **Повторное использование компонентов**: Хорошо структурированные компоненты (`SensorCard`, `ProcessChart`) делают интерфейс единообразным.

## 3. Слабые стороны

### 💾 Сохранность и целостность данных
*   **Риск In-Memory БД**: `sql.js` работает в памяти и сохраняет данные на диск по таймеру (30 сек) или при выключении. Сбой питания или аварийное завершение процесса приведет к потере данных текущей сессии. Это критично для длительных процессов, таких как брожение.
*   **Конкурентный доступ**: Блокировки файла SQLite могут стать проблемой при масштабировании или доступе нескольких процессов (хотя в текущей конфигурации маловероятно).

### ⚙️ Связанность логики процессов
*   **Логика на стороне фронтенда**: Большая часть критической логики процессов (например, шаги затирания, таймеры) находится в React-компонентах (`Mashing.jsx`).
    *   *Риск*: Если вкладка браузера закроется или зависнет, "логика" процесса остановится, даже если бэкенд продолжит нагрев. "Источник истины" для активного процесса должен быть на бэкенде.
*   **Захардкоженные значения**: Шаги и конфигурации иногда прописаны прямо в компонентах, вместо загрузки из БД/рецептов.

### 🛡 API и безопасность
*   **Валидация ввода**: Эндпоинты API не имеют строгой валидации (например, через `Zod` или `Joi`). Некорректный JSON может вызвать ошибку.
*   **Обработка ошибок**: Нет централизованного middleware для обработки ошибок в Express; полагаются на отдельные блоки `try-catch`.

## 4. Точки роста и план развития (Roadmap)

### Этап 1: Надежность (Backend-First Logic)
**Цель**: Сделать систему надежной, даже если UI ведет себя непредсказуемо.
1.  **Перенос состояния процесса на бэкенд**: Создать `ProcessManager` на бэкенде, который управляет конечным автоматом (Нагрев → Пауза → Кипячение). Фронтенд должен только *отображать* это состояние, а не управлять им.
2.  **Надежная база данных**: Миграция на `better-sqlite3` (если позволяют нативные модули) для прямой записи в файл, либо уменьшение интервала сохранения / принудительное сохранение при важных событиях.

### Этап 2: UX и масштабируемость
1.  **Движок рецептов**: Отделить выполнение рецепта от конкретной страницы 'Mashing'. Позволить создавать динамические шаги, определяемые данными рецепта.
2.  **Система уведомлений**: Улучшить интеграцию с Telegram, сделать её более настраиваемой для каждого пользователя/сессии. Добавить push-уведомления в браузере.

### Этап 3: Расширение аппаратной части
1.  **Конфигурация оборудования**: Создать UI для переназначения пинов/датчиков без изменения кода.
2.  **Автонастройка PID**: Реализовать алгоритм автонастройки (Autotune) для автоматического подбора коэффициентов Kp, Ki, Kd.

## Итог

Проект находится в **отличном состоянии** для прототипа/MVP. Код чистый, современный и хорошо протестирован. Основная область для улучшения — перенос **контроля процесса** с фронтенда (React) на бэкенд (Node.js) для обеспечения безопасности и надежности во время длительных варок.
