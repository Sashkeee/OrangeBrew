# План: CI/CD, Деплой на VPS и Авторизация (OrangeBrew)

Этот документ описывает стратегию перехода к профессиональному развертыванию приложения, выводу его в интернет и защите доступа.

## 1. CI/CD (Непрерывная интеграция и доставка)
Вариант с CI/CD (например, Github Actions) позволит автоматически обновлять систему при каждом написании нового кода.

### Пошаговый план внедрения:
1. **Создание репозитория**: Инициализация Git и загрузка кода в приватный репозиторий на GitHub.
2. **Настройка окружения**: 
   - Создание `.env` файла на целевом сервере (содержит ключи, пароли, порты).
   - Настройка `systemd` службы для автоматического перезапуска Node.js при сбоях и после обновления.
3. **Написание GitHub Workflow (`.github/workflows/deploy.yml`)**:
   - Триггер: `push` в ветку `main`.
   - Action будет подключаться к вашему серверу (Апельсинке или VPS) по SSH с использованием SSH-ключей.
   - Скрипт автоматизации выполнит:
     ```bash
     git pull origin main                 # Загрузка свежего кода
     npm ci --prefix backend              # Обновление зависимостей бэкенда
     npm ci --prefix frontend             # Обновление зависимостей фронтенда
     npm run build --prefix frontend      # Сборка React фронтенда
     sudo systemctl restart orangebrew    # Перезагрузка Backend сервера
     ```

## 2. Деплой на VPS / VDS
**Можно ли задеплоить на VPS?** Да! И это, пожалуй, самый стабильный и надежный вариант, если вы хотите иметь доступ к пивоварне с телефона из любой точки планеты.

### Как это будет работать (Архитектура):
* **VPS (Облачный сервер):** На нем будет крутиться Backend (Node.js) и база данных SQLite (рецепты, история). Там же будет раздаваться Frontend. Сервер имеет статичный (белый) IP и привязанный домен (например, `mybrew.com`).
* **Как подключить ESP32:** Ваша ESP32 дома подключается к обычной домашней Wi-Fi сети. В коде ESP32 (вместо локального IP 192.168.x.x) мы прописываем адрес вашего VPS (`ws://mybrew.com/ws`). 
* **Процесс:** ESP32 устанавливает постоянное *исходящее* WebSocket-соединение с VPS. Вам не нужно пробрасывать порты на домашнем роутере (NAT) или покупать белый IP от провайдера для дома. 
* **Пользователь (Вы):** Вы заходите на сайт `mybrew.com` с телефона по 4G. Сайт подключен к VPS. Вы нажимаете кнопку "Включить насос", сервер передает эту команду по открытому туннелю прямо на ESP32 дома.

### Плюсы и минусы:
* **Плюсы:** Абсолютная доступность везде, защита от DDoS силами провайдера/Cloudflare, не нужен OrangePi дома (нужна только плата ESP32 и обвязка реле), бэкапы БД средствами хостинга.
* **Минусы:** Требует стабильного интернета дома. Если провайдер отрубит домашний интернет, вы не сможете управлять пивоварней. 
* **Безопасность:** В коде ESP32 у нас УЖЕ реализован `Watchdog`. Если интернет пропадает больше чем на 15 секунд, ESP32 в целях защиты аппаратно отрубает ТЭН и все реле до восстановления связи или физического вмешательства.

## 3. Авторизация и Безопасность
Поскольку панель управления будет смотреть в интернет (на VPS или через туннели с Апельсинки), **авторизация обязательна**, иначе любой желающий сможет "покрутить" ваш ТЭН.

### Как лучше реализовать (Технический стек):
Лучший современный подход для связки React + REST/WebSocket API — это **JWT (JSON Web Tokens) + Bcrypt**.

**Пошаговый план:**
1. **База данных (SQLite):**
   - Добавление таблицы `Users` (id, username, password_hash, role).
   - При первом запуске бэкенда программа создаст админскую учетную запись с дефолтным сложным паролем, который нужно будет сменить.
2. **Backend (Node.js):**
   - Установка библиотек `jsonwebtoken` и `bcrypt`.
   - Создание POST-роута `/api/auth/login`. На входе логин/пароль, на выходе — сгенерированный JWT-токен.
   - Создание Middleware функции, которая будет проверять наличие правильного токена в заголовках `Authorization: Bearer <token>` для всех приватных API-запросов (рецепты, управление, история).
3. **Frontend (React):**
   - Создание полноценного экрана авторизации (Login Screen) в стиле SCADA, закрывающего доступ к приложению.
   - Сохранение полученного JWT токена в `localStorage`.
   - Axios Interceptors: автоматическое добавление токена к каждому HTTP-запросу. Если сервер отвечает `401 Unauthorized` (токен истек) — автоматический редирект на экран логина.
4. **Защита WebSockets:**
   - Для фронтенда: при открытии сокета первым сообщением отправлять токен авторизации. Если токен невалиден, сервер принудительно рвет сокет.
   - **Для ESP32 (Важно!):** ESP32 не может вводить логин и пароль. Для железа мы создадим статический и очень длинный `HARDWARE_API_KEY` (вшивается в прошивку ESP и в `.env` сервера). ESP32 будет подключаться к сокету и передавать этот ключ: `{"auth_key": "YOUR_SECRET_HARDWARE_KEY"}`.

> [!CAUTION]
> Без авторизации вывод системы в интернет категорически запрещен, так как это прямое управление силовым оборудованием (220В)!
